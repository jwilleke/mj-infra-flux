# Secrets Inventory

Audit of all `Secret` resources in the cluster. **Last refreshed: 2026-05-22** (covering the further #77 item 1 migration work).

The categorization tells you, for any Secret you find, **where its source of truth lives** — the most common question when investigating drift, doing a recovery rebuild, or onboarding a new agent.

## Categories

A Secret falls into exactly one of:

| Category | Source of truth | Action on cluster rebuild |
| --- | --- | --- |
| **Auto-generated** | Controller (cert-manager, k3s, k8s SA controller) | Re-issued automatically |
| **Helm-release state** | Helm itself (chart values + release history) | Re-created when the chart is reapplied |
| **In-repo (SOPS)** | `mj-infra-flux@HEAD`, decrypted by Flux's kustomize-controller via `flux-system/sops-age` | Re-applied by Flux on next reconcile |
| **Bootstrap** | Manual one-time apply — must be re-created out-of-band before Flux can do anything | Manual apply (chicken-and-egg) |
| **Hand-applied (drift)** | Operator memory — should be migrated to in-repo SOPS | Re-create or recover from backup |
| **Stale** | Nothing — orphaned, candidate for deletion | n/a |

## cert-manager ClusterIssuer drift (separate problem, surfaced 2026-05-22)

Discovered during `cloudflare-api-token` migration: live `ClusterIssuer/letsencrypt-{production,staging}` have a DNS-01 Cloudflare solver, but Git declares HTTP-01 only. Both carry `kubectl.kubernetes.io/last-applied-configuration` annotations, no Flux ownership labels — hand-applied at bootstrap.

**Root cause:** `apps/base/cert-manager/` is referenced *only in a comment* in `apps/production/kustomization.yaml` ("Infrastructure - Traefik ingress controller and cert-manager / These are in apps/base/ and work properly"), never as a resource. Flux has never reconciled cert-manager paths.

**Implication:** A fresh-cluster rebuild from `mj-infra-flux@HEAD` would NOT have DNS-01 working; certs requiring DNS-01 (e.g., wildcard or for services not externally reachable for HTTP-01) would fail to issue. HTTP-01 ones would still work.

**Out of scope for #77 item 1.** Tracked as a separate follow-up — to be filed as its own issue.

## Recovery procedure (if `deby` is rebuilt)

In order:

1. **Bootstrap age key**: restore `~/.config/sops/age/keys.txt` from the `/deby-backup/.../flux-sops-age.key` artifact, then `kubectl -n flux-system create secret generic sops-age --from-file=age.agekey=…`.
2. **`flux bootstrap`** the cluster against this repo. Once flux-system Kustomization reconciles, *all* in-repo SOPS Secrets are re-applied automatically (including `cloudflare-api-token` as of 2026-05-22).
3. **Re-create hand-applied Authentik trio** (`authentik/{authentik,authentik-postgresql,authentik-secrets}`) — until they're migrated, this is operator memory or external backup.
4. **Re-apply the live ClusterIssuer DNS-01 config** (see "cert-manager ClusterIssuer drift" above) — operator memory until the drift is resolved in Git.
5. **Confirm cert-manager re-issues TLS Secrets** by checking `kubectl get certificate -A` shows `Ready: True` for each. ACME rate-limits apply if many certs renew at once.
